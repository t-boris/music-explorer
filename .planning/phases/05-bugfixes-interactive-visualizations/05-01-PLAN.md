<task type="auto">
  <name>Task 1: Remove force-dynamic from client-side auth pages</name>
  <files>src/app/(auth)/dashboard/page.tsx, src/app/(auth)/practice/page.tsx, src/app/(auth)/progress/page.tsx, src/app/(public)/levels/[levelId]/lessons/[lessonId]/test/page.tsx</files>
  <action>
  Remove `export const dynamic = "force-dynamic";` from all four files.

  **Why this is safe:** These are all `"use client"` pages. The `force-dynamic` export is a Next.js server-side directive that prevents static generation and forces server-side rendering on every request. But client components already render on the client — the server only sends the initial JS bundle. Using `force-dynamic` on `"use client"` pages causes Firebase Hosting to invoke a Cloud Function for every page load, adding ~1-2s latency and consuming function invocations, for zero benefit.

  Without `force-dynamic`, these pages will be served as static shells (just the JS bundle) and hydrate on the client with auth state from the AuthProvider. The middleware still runs on every request to check auth cookies regardless.

  **Files to modify:**
  - `src/app/(auth)/dashboard/page.tsx` — line 3: delete `export const dynamic = "force-dynamic";`
  - `src/app/(auth)/practice/page.tsx` — line 3: delete `export const dynamic = "force-dynamic";`
  - `src/app/(auth)/progress/page.tsx` — line 3: delete `export const dynamic = "force-dynamic";`
  - `src/app/(public)/levels/[levelId]/lessons/[lessonId]/test/page.tsx` — line 17: delete `export const dynamic = "force-dynamic";`
  </action>
  <verify>npm run build passes. Build output shows /dashboard, /practice, /progress as ○ (Static) instead of ƒ (Dynamic). The test page may still be dynamic if it uses useParams — that's fine.</verify>
  <done>Removed unnecessary force-dynamic from 4 client-side pages, eliminating redundant Cloud Function invocations.</done>
</task>
