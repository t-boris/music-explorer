  <action>
**useSharedDashboard hook** (`src/hooks/use-shared-dashboard.ts`):
- `"use client"` directive
- Takes `targetUserId: string`
- Uses `useAuth()` to get current user
- On mount:
  1. Verify the current user has a connection to this target user (call `getMyConnections(currentUserId)` and check if targetUserId is in the list as a `fromUserId`). If not connected, set `error: "Not connected"`.
  2. If connected: fetch the target user's dashboard data using `getDashboardData(targetUserId)` — the same function used for the user's own dashboard. This works because security rules now allow any authenticated user to read user data.
  3. Also fetch the target user's recent recordings, test attempts, exercise completions, and practice sessions for the commentable items list.
- State: `data: DashboardData | null`, `loading: boolean`, `error: string | null`, `authorized: boolean`
- Returns `{ data, loading, error, authorized }`

For fetching commentable items, create helper functions in the hook or service:
- Reuse `getSessionHistory` from progress-service for practice sessions
- Query `users/{targetUserId}/recordings` for recordings (last 20)
- Query `users/{targetUserId}/testAttempts` for test results (last 20)
- Query `users/{targetUserId}/exerciseCompletions` for completions (last 20)

**Shared dashboard page** (`src/app/(auth)/community/[userId]/page.tsx`):
- `"use client"` directive
- Reads `userId` from route params
- Uses `useSharedDashboard(userId)` hook
- If not authorized: show "You don't have access to this dashboard. Ask them to share their invite link with you."
- If loading: spinner
- Layout mirrors the dashboard page but read-only:
  - Header: "{displayName}'s Dashboard" with back link to /community
  - StreakDisplay (read-only, same component)
  - ProgressOverview (read-only, same component)
  - Recent Practice Sessions list with CommentSection on each
  - Recent Recordings list with CommentSection on each
  - Recent Test Results list with CommentSection on each
- Each item card (session, recording, test) shows:
  - Item summary (title, date, score if applicable)
  - Expandable/collapsible CommentSection below it
- Use a simple accordion-like pattern: click "Comments (N)" to expand the comment section for an item
- Do NOT show QuickRecord or TodayPlan — those are personal action widgets, not relevant for viewing

**exercise-service.ts** — In `toggleExerciseCompletion`, after successful completion (not uncomplete), call:
```ts

```
We need displayName and photoURL — add them as parameters to toggleExerciseCompletion, or fetch from the user doc. Simplest: add optional `userDisplayName` and `userPhotoURL` params to the toggle function, passed from the hook which has access to auth user.

**test-service.ts** — In `saveTestAttempt`, after saving, call:
```ts
logActivity(userId, {
  type: "test_completed",
  title: `Completed test: ${data.testTitle} (${data.score}/${data.totalQuestions})`,
  metadata: { testId: data.testId, score: String(data.score), total: String(data.totalQuestions) },
  userDisplayName, userPhotoURL
})
```

Add optional userDisplayName/userPhotoURL params.

**practice-service.ts** — In `createPracticeSession`, after creating, call:
```ts
logActivity(userId, {

  userDisplayName, userPhotoURL
})
```
Add optional userDisplayName/userPhotoURL params.

**recording-service.ts** — Find the function that creates recordings. After creating, call:
```ts
logActivity(userId, {

})
```

For all activity logging calls: wrap in try/catch so failures don't break the primary operation. Activity logging is a non-critical side effect.

NOTE: The userDisplayName and userPhotoURL params should be optional with defaults. If not provided, the activity event still gets created but with "User" as display name. The calling hooks (use-exercise-completions, etc.) already have access to `useAuth().user` which has displayName and photoURL.
