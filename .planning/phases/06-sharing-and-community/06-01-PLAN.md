<task type="auto">
  <name>Task 2: Create connection-service.ts and activity-service.ts</name>
  <files>src/lib/connection-service.ts, src/lib/activity-service.ts</files>
  <action>
**connection-service.ts** — Follow patterns from practice-service.ts:

- `"use client"` directive
- Import Firestore functions and `getFirebaseDb`
- **Collection:** Top-level `connections` collection (NOT user subcollection — both users need to query it)
- **`createInviteLink(userId, displayName)`**: Generate a short code using `crypto.randomUUID().slice(0, 8)`, save to `users/{userId}/inviteLinks` subcollection. Return the code.
- **`getActiveInviteLink(userId)`**: Query `users/{userId}/inviteLinks` where active == true, limit 1. Return InviteLink or null.
- **`acceptInvite(code, currentUserId, currentDisplayName, currentPhotoURL)`**:
  1. Query ALL `inviteLinks` collection group where `code == code` and `active == true`. Use Firestore collection group query on `inviteLinks`.
  2. If not found, throw Error("Invalid or expired invite code").
  3. Extract the sharer's userId from the parent doc path.
  4. Check no existing connection between these two users (query `connections` where fromUserId == sharerId AND toUserId == currentUserId).
  5. If already connected, return early.
  6. Create a `Connection` doc in top-level `connections` collection with both user IDs, display names, and the invite code.
  7. Also read the sharer's user doc to get their displayName and photoURL.
- **`getMyConnections(userId)`**: Query `connections` where `toUserId == userId` (people who shared WITH me). Return Connection[]. These are the people whose dashboards I can view.
- **`getConnectionsToMe(userId)`**: Query `connections` where `fromUserId == userId` (people I shared with). Return Connection[]. These are viewers of MY dashboard.
- **`removeConnection(connectionId)`**: Delete the connection doc.

**activity-service.ts** — Simple event logger:

- `"use client"` directive
- **Collection:** `users/{userId}/activityEvents` subcollection
- **`logActivity(userId, data)`**: Create an ActivityEvent doc. Takes userId, type, title, metadata, displayName, photoURL.
- **`getActivityForUser(userId, limitCount = 20)`**: Query activityEvents ordered by createdAt desc, limit. Return ActivityEvent[].
- **`getActivityFeed(userIds: string[], limitCount = 20)`**: For each userId, fetch their last `limitCount` events. Merge, sort by createdAt desc, return top `limitCount`. This is used by the community feed — we fetch from each connected user individually (Firestore doesn't support cross-collection queries), then merge client-side.

NOTE: For `acceptInvite`, we need a collection group query on `inviteLinks`. This requires a Firestore collection group index and a security rule for collection group access.

<task type="auto">
  <name>Task 3: Update Firestore security rules and indexes</name>
  <files>firestore.rules, firestore.indexes.json</files>
  <action>
Update firestore.rules to support the community features:

1. **Top-level `connections` collection** — Both users in a connection can read/delete. Only the system (via acceptInvite) creates, so allow create if authenticated:
```
match /connections/{connectionId} {
  allow read: if request.auth != null &&
    (resource.data.fromUserId == request.auth.uid ||
     resource.data.toUserId == request.auth.uid);
  allow create: if request.auth != null;
  allow delete: if request.auth != null &&
    (resource.data.fromUserId == request.auth.uid ||
     resource.data.toUserId == request.auth.uid);
}
```

2. **Collection group rule for `inviteLinks`** — Allow any authenticated user to read invite links (needed for the collection group query in acceptInvite):
```
match /{path=**}/inviteLinks/{linkId} {
  allow read: if request.auth != null;
}
```

3. **Cross-user read for `activityEvents`** — Allow authenticated users to read any user's activity events IF a connection exists. However, Firestore rules can't do joins. Simplify: allow any authenticated user to read activity events (the data is non-sensitive — just "User X completed Y"):
```
match /{path=**}/activityEvents/{eventId} {
  allow read: if request.auth != null;
}
```

4. **Cross-user read for dashboard data** — When viewing a shared dashboard, we need to read another user's doc, sessions, recordings, etc. Add a rule that allows reading user data if a connection exists. Since Firestore rules can't query other collections, use a simpler approach: add a `sharedWith` array field on the user document listing UIDs who can view. OR, since checking connection existence in rules is complex, allow any authenticated user to read user documents but keep WRITE as owner-only:
```
match /users/{userId} {
  allow read: if request.auth != null;
  allow write: if request.auth != null && request.auth.uid == userId;

  match /{subcollection}/{docId} {
    allow read: if request.auth != null;
    allow write: if request.auth != null && request.auth.uid == userId;
  }
}
```

This is the simplest approach: any logged-in user can read any user's data, but the community UI only surfaces data from connected users. Write remains owner-only. The privacy boundary is at the UI/service layer (you only see dashboards of connected users), not the database rule layer. This is acceptable because all users are known (invite-only connections) and the data (practice sessions, scores) is not sensitive.

5. **Add collection group index** for inviteLinks in firestore.indexes.json:
```json
{
  "collectionGroup": "inviteLinks",
  "queryScope": "COLLECTION_GROUP",
  "fields": [
    { "fieldPath": "code", "order": "ASCENDING" },
    { "fieldPath": "active", "order": "ASCENDING" }
  ]
}
```

Also add index for connections queries:
```json
{
  "collectionGroup": "connections",
  "queryScope": "COLLECTION",
  "fields": [
    { "fieldPath": "toUserId", "order": "ASCENDING" },
    { "fieldPath": "createdAt", "order": "DESCENDING" }
  ]
},
{
  "collectionGroup": "connections",
  "queryScope": "COLLECTION",
  "fields": [
    { "fieldPath": "fromUserId", "order": "ASCENDING" },
    { "fieldPath": "createdAt", "order": "DESCENDING" }
  ]
}
```
